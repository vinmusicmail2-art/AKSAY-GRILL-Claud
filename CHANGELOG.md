## [2 мая 2026] — Рефакторинг, документация, карта архитектуры

**Коммит:** `4aa888c` · **Файлы:** `mailer.py`, `app.py`, `routes_public.py`, `replit.md`

### Debug-helper — исправленные баги

- **`mailer.py`**: дублированный `from db import SessionLocal` внутри функции
  `_get_smtp_password_from_db()` — убран (модуль уже импортирован на верхнем уровне).
- **`mailer.py` `_send_smtp()`**: функция `smtp_status()` вызывалась внутри `_send_smtp()`
  хотя та уже имела `cfg` от своего же вызова `_get_smtp_config()` — двойной вызов конфига
  на каждую отправку письма. Исправлено: проверка missing-полей теперь идёт напрямую из `cfg`.
- **`mailer.py` `_send_notification_async()`**: самодельный класс `_Shim` с `setattr`-циклом
  для преобразования `dict → object` заменён стандартным `types.SimpleNamespace(**data)`.
- **`app.py`**: при отсутствии переменной окружения `SESSION_SECRET` приложение молча
  использовало небезопасный fallback `"dev-secret-change-me"`. Теперь пишется предупреждение
  в лог.
- **`app.py` `load_user()`**: задокументирован latent risk — возврат detached ORM-объекта
  после закрытия сессии (безопасно пока у `Admin` нет lazy relationships).

### Simplify — упрощение повторяющегося кода

- **`mailer.py`**: извлечен хелпер `_guarded_send(format_fn, req, base_url)`.
  Все 5 функций `send_*_notification()` дублировали одинаковые 4 строки:
  `_get_recipient_and_toggle()` → проверка `enabled` → проверка `recipient` → `_send_smtp()`.
  Теперь каждая из них — одна строка вызова `_guarded_send(...)`.
  Удалено ~20 строк повторяющегося кода.

### Docs-generator — добавленные docstring'и

- **`app.py`**: `init_db`, `inject_site_texts`, `load_user`, `_client_ip`,
  `_is_safe_next`, `_safe_referrer`, `add_cache_headers`.
- **`mailer.py`**: `_get_smtp_password_from_db`, `_send_smtp`, `_guarded_send`,
  `_send_notification_async`, все `send_*_notification`.
- **`routes_public.py`**: `contact`, `home`, `business_lunch`, `catering`,
  `events`, `order_delivery`, `quick_request`.

### Архитектурная документация

- **`replit.md`** полностью переписан: карта архитектуры (top-down дерево модулей),
  таблица всех маршрутов Admin + Public, раздел «Key Patterns», раздел
  «Known Technical Debt» (inline-миграции в `init_db`, context_processor на каждый запрос,
  хардкод email в `send_contact_question`).

## [Апрель 2026] — E-mail уведомления и бизнес-ланчи

### Добавлено: e-mail уведомления (`mailer.py`)

**Файлы:** `mailer.py`, `models.py` (SiteText-ключи), `app.py`,
`templates/admin/email_settings.html`, `templates/admin/dashboard.html`

- Новый модуль `mailer.py` — отправка e-mail через стандартную библиотеку
  `smtplib`; поддержка порта 465 (SSL) и 587 (STARTTLS).
- Все уведомления отправляются в фоновом потоке (`threading.Thread`) —
  не блокируют HTTP-ответ.
- Безопасный fallback: если SMTP не настроен или отключён, заявка
  сохраняется в БД, ошибка только логируется.
- Два управляющих ключа в `site_texts`: `notify_email_recipient` (адрес
  получателя) и `notify_email_enabled` (вкл/выкл).
- Функции: `send_order_notification` / `send_order_notification_async`
  (бизнес-ланчи), `send_test_email`, `smtp_status`.
- Подробные HTML-письма в фирменном стиле (терракотовый header, таблица
  полей, кнопка «Открыть в админке»).
- Страница `/admin/email-settings`:
  - статус SMTP (настроен / не настроен, без раскрытия пароля),
  - поле адреса получателя,

## Схема базы данных (актуальная, май 2026)

| Таблица | Назначение |
|---|---|
| `admins` | Учётные записи администраторов (bcrypt, Flask-Login) |
| `login_logs` | Журнал попыток входа — IP, UA, успех/отказ (152-ФЗ) |
| `site_texts` | Редактируемые тексты сайта (key-value) |
| `menu_categories` | Категории меню (видимость, порядок, показ в навигации) |
| `menu_items` | Блюда каталога (цена, вес, описание, изображение) |
| `delivery_orders` | Заказы из корзины на сайте (JSON состав + processed_by) |
| `quick_requests` | Быстрые заявки с кнопки «Заказать» на главной |
| `business_lunch_orders` | Заявки на корпоративные бизнес-ланчи (+ processed_by) |
| `catering_requests` | Заявки на выездной кейтеринг (+ processed_by) |
| `hall_reservations` | Заявки на бронирование зала (+ processed_by) |

Все четыре таблицы заявок содержат поля аудита:
`is_processed` (bool), `processed_at` (datetime), `processed_by` (username), `created_at`, `ip_address`.
